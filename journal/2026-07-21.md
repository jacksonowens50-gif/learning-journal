## What I learned
- **List of dicts** is the shape of table data. Each dict is a row, each key is a column. Same as a SQL result set. This is also what every API response looks like.
- **Dict lookup** — `some_dict[key]` returns the value. Brackets on a list take a position number, brackets on a dict take a key name.
- **Dict as a lookup table** instead of a giant if/elif chain (NORMAL_SIDE maps account type to debit/credit). Adding a new type = adding one line of data, not editing logic.
- **List of dicts vs flat dict**: use a list of dicts when you'll loop through everything, a flat dict when you'll look up one thing by key.
- **Accumulator pattern** — `total = 0` before the loop, `total += x` inside. Start at 0 for sums, start at 1 for products.
- **Build-a-list pattern** — empty list before the loop, `.append()` inside, `return` after. The `return` must be OUTSIDE the loop (made this mistake, same bug as `has_lucky_number`).
- **Counter/group-by idiom** — empty dict, seed a key at 0 the first time you see it, then `+=`. That's `SUM() GROUP BY` in Python.
- **List comprehensions** — `[n**2 for n in range(10)]`. Read the `for` first (the loop), then the front (what you collect), then any `if` at the end (the filter).
- **enumerate()** gives you index AND item at once, vs `range(len())` which gives only the index.
- **try/except** — `try` runs the risky code, `except SpecificError` catches it and returns a fallback instead of crashing. Catch specific errors, not bare `except`. Can catch multiple with a tuple.
- **f-strings** — the `f` prefix makes `{}` a placeholder. After the colon is formatting: `<22` left-aligns, `>12,` right-aligns with commas. Text left, numbers right = columns line up.
- **`"\n".join(lines)`** — glues a list of strings into multi-line text. Separator on the left, list on the right.
- **Helper functions** — split a big job into small ones. `hand_total()` for blackjack, `word_search()` reused inside `multi_word_search()`, `normal_side()` and `sum_balances()` reused inside `format_report()`.

## Question / Stuck points
- Kept mixing up which function needed which input — `filter_by_type` doesn't touch NORMAL_SIDE, that's only for the report. Lesson: each function does ONE job.
- Variable shadowing bug in the Mario Kart exercise — inner loop reused `i` from the outer loop and silently overwrote the index. Never reuse a loop variable name in a nested loop.
- `=` vs `==` — one assigns, two compares.
- `safe_balance` returning 0.0 is ambiguous: can't tell an error from a real zero balance. Real code should print a warning too.

## Notes
### The SQL mapping that made this click
- `filter_by_type()` = `WHERE account_type = ?`
- `sum_balances()` = `SUM(balance)`
- `group_totals()` = `SUM(balance) GROUP BY type`
- Composing them: `sum_balances(filter_by_type(accounts, "asset"))` = SUM WHERE

### Loop mental model
"Do this block once for each thing in the collection." The loop variable is a box that gets refilled each pass. Use `for x in list` when you only need the item; use an index only when position matters (like comparing neighbors).

### Why `range(len(x)-1)` when comparing neighbors
There's always one fewer PAIR than there are items. 3 meals = 2 adjacent pairs. Without the -1, `x[i+1]` runs off the end of the list.
